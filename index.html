数据分离与运行时查询

发表于 2026-03-17

数据分离与运行时查询

序

由于L1缓存弥足珍贵，故而我们在设计 Component 时往往又会回到OOP老生常谈的一个原则：高内聚，低耦合。
实际场景中一个 Entity 往往拥有多个 Component ，当然也不排除单一的存在。
所以本篇主旨就是关于 Component 的设计理念与运行时查询的原则。

数据分离 - 冷热隔离

Component 与结构性突变的抉择

结构性突变是高性能并发中的红线，会产生幽灵数据，强制同步等一系列棘手的问题。
设计原则应该是 尽可能地稳定实体的 Components 来尽可能的减少结构性突变的发生。

单一 Entity 上应该有可满足需求的多个组件，每个组件需要复合设计标准且尽可能地内聚

在初始化时一次性支付所有的开销，但在使用时就不要再去增删组件，而是通过内部标记来表示当前组件是否激活。
这是一个标准的空间换时间方案，想必读者一目了然。
关于结构性突变的内容请查阅相关Blog，在此不赘述。

单一Component 设计理念

如果一个Entity 确实只需要一个Component就能够表达，那么设计方案其实就相对的简单。
使用我们的自定义结构布局，结合位压缩算法，就可以使单一的 Component 足够小巧的同时也可以表达足够丰富的数据。
要点在于冷热分离，假设我们使用一个 ulong 类型的结构来表示。
我们需要将频繁处理的数据，例如 Position 和 Rotation 放在一起，并集中在低位内存中。
而将不经常处理的数据，例如标记位等数据，放置在高位中。做一个简单的冷热分离。

在 CPU 层面，ulong（8字节）是内存访问的最小原子单位之一。所以无论怎么放这些数据，他们都会被加载进入缓存。
在物理层 Memory（存储）并没有什么区别，而在 ALU（计算）层由于数据高内聚会带来的指令优化，但在此不做过多讨论。

多 Component 复合 Entity 设计方案

多组件复合的情况下会涉及到更多的设计考量。但在这里我们只讨论一下它的好处。
数据吞吐量
现实场景中，我们可能只是更改实体部分的属性，我们将这部分称之为热数据，它们总是频繁出入L级缓存。
所以分离数据为多个 Component 就可以降低无效数据从内存写入L1缓存，进而提高有效数据的吞吐量。

举个例子：试想若有 X 个实体。拥有单一 Component 结构，大小为32字节。而其中16字节为需要频繁修改的热数据。
若单一数据结构下，每次都需要将32字节大小的结构写入缓存。
若使用多 Component 结构，拆分32字节为2个16字节的 Component 组合形式。
则除非全量更新，每次搬运的数据量则会减少一半。

实际上，即使是全量更新，在稳定的环境下（即不产生数据回环上限导致的地址冲突）多 Component 的设计方案，速度与单一结构基本可以持平。
只是需要更多的维护成本。但显然这不是CPU考虑的事情 (#`Д´)ﾉ

补充

与直觉相反，Unity官方提供的方案中更加推荐使用Tag（结构性突变）。
对于上文的Bit标记位结构来说，无论修改了多少的数据，都需要将全部数据拉入缓存。

对比：

  • 使用 Tag (结构突变)：EntityQuery 在底层直接跳过不含该 Tag 的整个 Chunk。CPU 根本不碰那些没被冻结的实体内存。
  • 使用 Internal Flag：Job 必须把所有实体的内存都拉进 L1 缓存，然后在循环里用 if 或 math.select 挑出需要的。

这一点没有绝对对错，按需设计即可。

L1 Cache & 8-Way Set Associative

发表于 2026-03-17

L1 Cache & 8-Way Set Associative

序

一篇关于 32KB L1d 缓存架构下 L1 Cache 相关设计以及特性的探究笔记。不涉及专业课本知识。
解释了内存布局设计以及分批策略的缘由。

物理结构 (Physical Sturct)

L1缓存的物理概念

注意区分 L1缓存(L1 Cache) 与 缓存行(Cache Line) 的概念，它们完全是两码事，但是彼此之间相互构成。
在 32KB L1d 缓存架构下，物理层级被划分为 64 个组 (Sets)，每个组包含 8 个行(Ways/Lines)。

上文所述的物理参数如下：

  • L1 数据缓存容量: 32 KB (32,768 字节)。
  • 缓存行 (Cache Line): 64 字节 (最小存取单位)。
  • 总行数: 512 行 (32768 / 64)。
  • 组关联 (Associativity): 8-路 (8-Way)，每 8 行组成一个 Set (组)。
  • 组总数: 64 个 Sets (512 / 8)。

通俗来说

  • 每 8 个bit组成了一个字节
  • 64 个 8-bit 组成了一个 Cache Line
  • 每 8 个 Cache Line 为一组，总计 64 组横向排列在一起，组成了 L1 Cache

此处缺少一张示意图

物理映射逻辑 (Mapping Logic) & 4KB回环 & 地址冲突

Mapping Logic
CPU预存器将数据从内存搬入L1缓存时，需要遵循以下规则：
内存地址到缓存组的映射由地址位决定

  • Offset (6 bits): $Addr[5:0]$，决定缓存行内的字节偏移（0-63）。
  • Index (6 bits): $Addr[11:6]$，决定所属组（0-63）。公式：$SetIndex = (Addr \gg 1) \ \& \ 0x3F$。
  • Tag (Remaining bits): 剩余高位地址，用于标识数据的物理归属。

其实表述的有点复杂了，简单的来说 索引计算: SetIndex = (地址 / 64) % 64

4KB回环 & 地址冲突
基于上述物理映射逻辑，不难发现两种情况：

  • 单一连续数据队列长度超过 4KB 。
  • 多条连续数据首地址为 4KB 倍数。

上述两种情况将会导致数据竞争同一个 Set ，但是单一 Set 最大 Cache Line 数量为8。
当同一 Set 内竞争的并发流超过 8 路时，触发 LRU 算法强制踢出旧行，导致流水线停顿 (Stall)。
进而导致性能下降。

应对策略

内存布局准则 (Layout Rules)

其实在前置的内容中写到过关于显式内存布局的准则。这些准则的由来相当一部分的原因上就是源自此处的物理特性。
详细的设计不在此赘述。

16KB 准则

基于上述小节可知，将单 Job 活跃数据量限制在 16KB (256 行)，每组 Set 可留出4个空位，进而有效缓冲 Set 竞争问题。

单流原则与多流处理

单流设计

每个 JOB 只对单一的组件（Component）进行处理。由于线性访问，且内存连续的机制以及内存墙的物理特性，可以有效的避免内存地址竞争。
单一 Component 的设计只需要遵循 Size 为64的因子即可。

多流设计

实际场景往往复杂的多，单流的设计更多的只适用于教学或者简单场景。
多组件在单一Job中操作：
多流设计遵循原则：多个 Component 之间综合也应该保持为 64 的因数设计大小
确保多个 Component 数据被预存器搬入L1缓存时，它们依旧可以填满物理内存空间而不产生越界与跨行。
他们可能是交替写入 Set 也有可能是顺序写入，但无论如何，依旧刚好可以对齐物理空间。
进而有效避免由不对称结构或者跨行导致的等一系列延申性能问题。

总结

在物理视角下 CPU 硬件层级不识别 C# 类型系统，也并没有要求指出所有的数据结构不能够跨越缓存行。
只是同样的数据计算量下，跨行或者非对齐结构，CPU需要更多的指令和额外的操作去完成。
其实也揭示了内存墙的特性之一：现代CPU在计算性能上已经算的足够快，大部分时间都是在等待数据从内存搬运进缓存

前导

让我们理一下概念逻辑：

在理想的情况下，每帧中主线程 CPU 只需要收割与结算上一帧的数据和下发新一轮并行指令。

基于完全二叉树最小堆重力索引池

发表于 2026-03-05

序

在ECS的情景下，对于 Entities 的生成与回收会对性能产生巨大的波动。
对象池的设计是正是解决此类问题的优解。
但是对于传统的对象池，在大量存取之后，其内存分布将会变得无序且极度分散，但是在高性能，或高并发的场景下需要尽可能的使得物理内存聚集连续
故有此篇。

设计理念

基于最小堆算法实现，如果读者对这个技术十分了解，则本篇已无继续阅读的必要。
由于Unity的限制，本篇依旧使用C#设计。

合法堆要求

  • 数据结构为完全二叉树。
  • 完全二叉树中父节点必须小于或者等于其左右子节点。

容器设计 - NativeList

我们需要一个容器类来存放暂时不用的数据。这个容器应该符合先进先出的原则，并且能够灵活的操作。
在这个场景下 NativeList 是较为贴合需求的选择。
在 NativeList 中，我们只要按顺序 Add 和 RemoveAt(last)，就可以保证整体的结构是一个标准的完全二叉树结构。
同时也能够自由的操作容器中的数据。
注意，由于使用的 NativeList ，所以类型只能够是值类型。

NativeList 中的性能深坑

如果读者有过性能方面的了解，想必知道 List 与 数组 下移除首位的性能开销。
NativeList 虽然也是 List ，但是由于为了保持内存连续的结构，在破坏结构后需要进行排序对齐。
如果我们移除了NativeList中的首位，则需要对后续的所有位排序。

在连续内存中，移除首位都是极其昂贵的操作。

出池队尾填充

在需求上我们需要拿到头部数据的同时，也不想支付昂贵的排序开销操作。


我们拿出了首位的数据，同时将队尾的数据写入首位中，然后将队尾的数据弹出。
这样我们就避免了因为移除首位导致的内存断层，进而避免了触发 NativeList 的排序。
同时，我们也破坏了上述的合法堆的结构要求，整个堆变成了非法堆。
但是问题变得更好解决，对于 CPU 而言，完全二叉树排序是一个非常简单又轻松的工作。
我们的问题从 连续内存断层后的数据移动 变为了 将错乱的完全二叉树合法化 。
具体的实现我们在下文中探讨。

“重力索引” - 修复破碎的非法堆

上文的操作中，我们已经知道，当前堆物理上是紧凑的，但是逻辑上是非法的。
在出池的修复逻辑下，问题的关键在于寻找子节点中的小值是否合法，即大于或等于父节点。

数学复杂度公式 (The Computational Bound)
对于一个拥有 $N$ 个元素的完全二叉树：

  • 最大交换次数 ($H$)：$\lfloor \log_2(N) \rfloor$
  • 对于 $N = 1024$：
    • 树的高度为 10 层。
    • 每层工作量：需要进行 2 次比较（子节点互比、父节点与最小子节点比）和 1 次交换。
    • 结论：即使最极端的最大索引，最多也只需要 20 次比较 和 10次交换。

入池的逻辑与出池基本类似，将数据加入队尾，同时通过算法恢复堆的合法结构。


数学复杂度公式 (The Computational Bound)
对于一个拥有 $N$ 个元素的完全二叉树：

  • 最大交换次数 ($H$)：$\lceil \log_2(N + 1) \rceil - 1$
  • 对于 $N = 1024$：
    • $2^{10} = 1024$，所以最大高度约为 10 层。
    • 结论：即使归还的是最末尾的一个最小索引，它最多也只需要 10 次 比较和 10 次 交换就能到达堆顶。

总结

以上，无论我们在出池或者入池时，都保障了堆的合法性。
不仅获取了头部数据，也避免了因为连续内存断层导致的庞大的性能开销。
至于初始化部分和运行时逻辑健壮性在此不过多讨论，依据具体需求实现即可。

尾记

至于为什么命名”重力索引“？我们以出池为例。
如果读者尝试推导这个计算过程就不难发现，这个算法的流程其实就是将末尾的数值放在二叉树顶部，然后逐步将其”踢回“属于它的位置。
就像一个石头从山峰上滚下去。

基于位运算的无分支门控注入与变体探究

发表于 2026-03-01

Date: 2026-03-01
Author: Errial
Tags: #UnityDOTS #HighPerformance #Branchless #16KB-L1

序

最近在设计高性能计算时遇到了性能需求，需要尽可能少的产生分支预测。
虽然函数库 Unity.Mathematics，其内部提供了math.select等相关高性能计算的API接口。
问题在于这些API即使反编译看起来也很晦涩难懂，所以自己实现了一版的门控逻辑。

一、 基本逻辑与实现

uint gate = (uint)(((long)rawValue - (long)(mask + 1)) >> 63);
是的，代码只有一行。其实非常的简单，但是涉及到的知识点较多，我们逐一分析。

基本原理与mask + 1 ？

rawValue是传入的原始值，Mask是标准模。理论上来说如果说一个数值大于Mask来说的话即非法。
所以方案就跃然纸上，使得两者相减，必然会获得三个数学结果
大于0，小于0，等于0。
mask + 1的意义就在于是否需要设计包含0的情况，因为0在shift后会产生“关门”的结果。
这里我们需要包含0，故而将mask扩大1，使得rawValue与mask相等时也会产生负数。

逻辑运算与计算运算以及掩码扩散机制

注意，笔者在进行算数相减前先做了一次类型转换，强制转为了long。注意，这是必要的。
显而易见，如果原类型是无符号类型，我们是无法获得负数的，此时执行的是纯粹的逻辑运算。
而如果是有符号类型，对于存在符号位的情况下，会进行运算计算，继而产生我们所说的掩码扩散机制。

将结果向右移sizeof(type) - 1位数。
如果读者的计算机基础尚可的话，就会明白，此时得到的结果如下:

  • 预期类型1：0xFFFF….FFFF
  • 预期类型2：0x0000….0000

门控的实现

通过上面的计算，我们得到了一个有符号类型的二进制数effectiveMask，全为1或者全为0。此时的逻辑已经接近于bool类型了。
然后在后续的计算中，就可以使rawValue & effectiveMask。来控制写入是否生效。

二、 双向选择网关 (The Select/MUX Gate)

基于上述的基本理论，我们可以利用我们的计算原型gate做更多的事情。

  • 代码实现：uint result = (A & gate) | (B & ~gate);
  • 物理解析：
    • 当 gate 为全 1：结果是 (A & 11..1) | (B & 00..0) -> A。
    • 当 gate 为全 0：结果是 (A & 00..0) | (B & 11..1) -> B。

基于上述功能，可以快捷的实现A或者B的选择。

三、 增量注入网关 (The Accumulator Gate)

一种快速的增量式写法

  • 代码实现：damage += (10 & gate);.
  • 解释：将“逻辑判断”转化为“加法中的零因子”。
